# Session, partition, userData, and proxy APIs — verified against Electron 43.0.0

**Date:** 2026-07-01
**Pinned version:** electron@43.0.0 / Chromium 150.0.7871.46
All method signatures below checked against `https://www.electronjs.org/docs/latest/api/session` (the "latest" docs track the current major release line, cross-checked for no v43-specific deprecation in `breaking-changes.md`).

## 8. `session.fromPartition('inmemory-session', { cache: false })` (no `persist:` prefix)

**Verified: TRUE — API and semantics still current, but the `cache` option means something narrower than the plan implies.**

- `session.fromPartition(partition[, options])` is unchanged in Electron 43. Omitting the `persist:` prefix on the partition name is still exactly what makes a partition **in-memory-only** (not backed by a disk directory under `userData/Partitions/`). This part of the claim is correct and load-bearing — keep it.
- The `cache` option's actual documented semantics: "Whether to enable cache. Default is `true` unless the `--disable-http-cache` switch is used." **This only toggles the HTTP disk/memory cache setting for that session** — it is not a general "make this session non-persistent" flag. The memory-only guarantee comes entirely from the missing `persist:` prefix, not from `cache: false`. Passing `cache: false` here is redundant with (but harmless alongside) the global `--disable-http-cache` switch from `command-line-switches.md` §1.
- Source: https://www.electronjs.org/docs/latest/api/session
- No code change required, but the code comment in `main.ts`/`main.js` should be corrected to say "cache:false is redundant with the global disable-http-cache switch, not what makes this session memory-only" so a future contributor doesn't remove the missing `persist:` prefix thinking `cache:false` alone covers it.

## 10. `app.setPath('userData', '/dev/shm/...')`

**Verified: TRUE with a real timing caveat the plan should call out explicitly.**

- `app.setPath(name, path)` still exists with the same signature: "Overrides the path to a special directory or file associated with name. If the path specifies a directory that does not exist, an Error is thrown." **Important, easy-to-miss detail confirmed in the current docs: the directory must already exist, or `setPath` throws** — `/dev/shm/amnesia-browser-<pid>` must be created (`fs.mkdirSync(..., {recursive: true})`) _before_ calling `app.setPath`, not after. The plan's snippet in Section 4.2 does not create the directory first — this needs a code fix, not just a version-compat note.
- Timing: Electron's own docs explicitly warn that some sub-paths (they call out `sessionData` specifically, which is the default `userData` sub-directory Chromium's cookies/cache live under) must be overridden **before the `ready` event fires** — matching what the plan already assumes (bootstrap flags run before window creation), so no change needed there, just confirm this must be the very first thing in `main.js`, ahead of any `session.fromPartition()` calls too, since creating a session before `userData` is redirected can cause Chromium to lazily cache the old path for that session's internal prefs.
- Source: https://www.electronjs.org/docs/latest/api/app
- **Net verdict:** API/timing assumption is correct; the plan's code sample is missing the pre-creation of the tmpfs subdirectory, which will throw `Error: ... does not exist` on Electron 43 if copied as-is. Flag this to whoever writes `main.ts`.

## 11. `session.clearStorageData()`, `session.clearCache()`, `session.clearAuthCache()`

**Verified: TRUE — all three current, all return `Promise<void>`, none deprecated.**

- `clearStorageData([options])` → `Promise<void>`. Options object supports `origin` and `storages` (list of storage types like cookies, filesystem, indexdb, etc.).
- `clearCache()` → `Promise<void>`. Clears the session's HTTP cache.
- `clearAuthCache()` → `Promise<void>`. Resolves when the HTTP auth cache is cleared.
- Source: https://www.electronjs.org/docs/latest/api/session
- **One real breaking change to be aware of (Electron 42.0, per `breaking-changes.md`):** the `quotas` option to `clearStorageData()` was removed because it was removed from upstream Chromium; `36.0` earlier removed the `syncable` quota type. The plan's example calls `clearStorageData()` with no options at all, so it is unaffected — but if implementation code later adds an options object with `quotas`, it will silently be ignored/invalid. Worth a code comment.
- All three already return Promises in v43 as the plan assumes (`await ses.clearStorageData()` etc. in Section 4.3 is correct usage) — no change needed to the cleanup routine's await-chain logic.
- Source: https://www.electronjs.org/docs/latest/breaking-changes

## 22. `ProxyConfig.proxyRules` syntax for SOCKS5, fail-closed fallback, and bypass rules — verified for ADR 0007

**Date:** 2026-07-05. Owner approval for Tor/SOCKS work given; verifying before implementation per CLAUDE.md's network-switch re-verification rule and ADR 0007 decision 3/4.

- **`proxyRules` grammar**, per `https://www.electronjs.org/docs/latest/api/structures/proxy-config`: `proxyRules = schemeProxies[";"<schemeProxies>]`, each `schemeProxies` either scheme-specific (`"http=foopy:80;ftp=foopy2"`) or a single default applying to all schemes (`"foopy:80"`). A proxy entry can itself be a **comma-separated failover list**: the docs' own example is `"foopy:80,bar,direct://"`, described as "Use HTTP proxy foopy:80 for all URLs, failing over to bar if foopy:80 is unavailable, and after that using no proxy." **This confirms ADR 0007 decision 4's fail-closed design is correct by construction, not just by omission**: failover (including a `direct://` escape hatch) only exists if the proxy string explicitly contains a comma-separated list. A bare `socks5://host:port` — no commas, no `direct://` — has no fallback for Chromium to fail open to; an unreachable proxy fails the navigation (`ERR_PROXY_CONNECTION_FAILED`-class error) instead. The implementation must never construct a comma-joined `proxyRules` string for Tor mode — a single `socks5://host:port` value is both sufficient and load-bearing for the kill-switch.
- **SOCKS5 scheme prefix**: the docs' own examples show `socks4://foopy` for SOCKS v4; `socks5://` is the documented sibling scheme prefix Chromium's proxy-rules parser recognizes for SOCKS version 5 specifically (distinct from bare `socks://`, which historically maps to SOCKS4 for backward compatibility in Chromium's proxy config parser — use `socks5://` explicitly, never the bare `socks://` alias, to avoid silently downgrading to SOCKS4's pre-resolved-IP-only behavior).
- **DNS-via-proxy for SOCKS5 vs SOCKS4** (ADR 0007 decision 3): official Electron docs do not document this at the wire-protocol level. This is standard SOCKS behavior (RFC 1928 defines a domain-name address type SOCKS5 clients use precisely to let the proxy resolve hostnames; SOCKS4 has no such address type and requires a pre-resolved IPv4 address), and Chromium's SOCKS5 client implementation is understood to use it — but per this project's verify-,-don't-assume standard, this is **empirically confirmed by the e2e test harness** (`tests/e2e/tor.spec.ts`), not asserted from documentation alone: the hand-rolled test SOCKS5 server logs the literal address-type byte and value from each CONNECT request, and the test asserts a domain-name arrives, never a resolved IP.
- **`proxyBypassRules` default / implicit localhost bypass**: Electron's structure docs describe the bypass-rule pattern language (hostnames, domain suffixes, IP literals, CIDR ranges, and a `<local>` token for local addresses) but do not state whether Chromium bypasses loopback addresses by default when this field is left empty. Chromium source (`net/base/proxy_bypass_rules.cc`, `net/proxy_resolution/proxy_bypass_rules.cc`) was not reachable in this environment to confirm directly (repeated 404s / fetch failures against `chromium.googlesource.com` for the exact path). **Resolved empirically instead**, consistent with this project's methodology: the e2e test harness includes a specific case that points a tab at a `localhost`-served page while Tor mode (with the fake SOCKS5 test proxy) is enabled and `proxyBypassRules: ''`. Confirmed result: the request reaches the local server **directly**, never the fake proxy — Chromium bypasses loopback by default regardless of an empty `proxyBypassRules`. See `tests/e2e/tor.spec.ts` and `docs/threat-model.md`'s network-observer row.
- Source: https://www.electronjs.org/docs/latest/api/structures/proxy-config (proxyRules grammar, SOCKS4/SOCKS5 scheme examples, proxyBypassRules pattern types)

## 23. `app.configureHostResolver` shape and repeat-call behavior — verified for ADR 0010

**Date:** 2026-07-05. Verifying before implementing the DNS-over-HTTPS toggle, per CLAUDE.md's re-verification rule for network-facing APIs.

- **Confirmed present on electron@43.0.0** via the pinned `node_modules/electron/electron.d.ts`: `app.configureHostResolver(options: ConfigureHostResolverOptions): void`, where `ConfigureHostResolverOptions` is `{ enableBuiltInResolver?: boolean; enableHappyEyeballs?: boolean; secureDnsMode?: 'off' | 'automatic' | 'secure'; secureDnsServers?: string[] }`. The type's own doc comment states Electron's un-configured default is `secureDnsMode: 'automatic'` (opportunistic DoH upgrade when the OS resolver advertises support) and that the method "must be called after the `ready` event is emitted."
- **Repeat-call behavior is undocumented** — the doc comment says nothing about whether a second call is honored, ignored, or an error. Not assumed either way: verified empirically with a standalone Electron script (`app.whenReady().then(() => { app.configureHostResolver(...) })` called four times in sequence with different `secureDnsMode`/`secureDnsServers` values). **Result: every call succeeded with no exception thrown**, in both directions (automatic → secure → secure with a different server list → back to automatic). This is the basis for this app's toggle design: `applyHostResolver()` (`src/main/index.ts`) is called once at startup and again on every provider change, exactly like `session.setProxy()` is re-called for Tor (ADR 0007) — no relaunch is required to change DNS provider mid-session.
- **Not verified**: whether a changed `secureDnsMode`/`secureDnsServers` actually takes effect for in-flight or soon-after DNS lookups at the network-service level (i.e., real packet-level proof that queries go to the new provider). This sandboxed environment has no root/netns access for packet capture, and this project does not stand up a live HTTPS DoH mock server for the same reason ADR 0007 didn't add certificate-bypass flags for a mock Tor daemon — it would require weakening TLS validation in a way that must never ship. See `docs/threat-model.md`'s DNS row for the honest scope of what is and isn't asserted in CI versus what a maintainer should manually verify (e.g. `sudo tcpdump -i any port 53` while toggling the setting) before relying on this feature.
- Source: `node_modules/electron/electron.d.ts` (pinned electron@43.0.0), plus the standalone verification script described above (not committed — a throwaway check, not a needed test fixture).

## 24. `ProxyConfig.proxyRules` for HTTP/HTTPS proxy schemes, DNS-at-proxy, and scheme-aware probes — verified for ADR 0012

**Date:** 2026-07-06. Generalizing ADR 0007's SOCKS5-only proxy to any user-supplied proxy scheme (HTTP/HTTPS alongside SOCKS5). Verifying before implementation per CLAUDE.md's network-switch re-verification rule.

- **`proxyRules` scheme prefixes for HTTP/HTTPS proxies**, per the same Electron `structures/proxy-config` docs used in §22: the grammar's per-scheme proxy value is `[<url-scheme>"://"]<proxy-host>[":"<proxy-port>]`, and the docs' own examples use bare `foopy:80` and `http=foopy` forms. The **proxy** URL-scheme prefix (`socks5://`, `http://`, `https://`) states how the browser reaches the proxy itself and is independent of the destination URL's scheme. `http://host:port` and `https://host:port` are the documented forms for an HTTP and an HTTPS (TLS-to-the-proxy) proxy respectively. Confirmed the `ProxyConfig` interface shape is unchanged on the pinned electron@43.0.0 via `node_modules/electron/electron.d.ts` (`proxyRules?: string`, `proxyBypassRules?: string`, `mode?`, `pacScript?`). The **fail-closed invariant is scheme-independent**: a bare `http://host:port` (no comma, no `direct://`) has no fallback entry for Chromium to fail open to, exactly as the SOCKS5 case in §22 — so the kill-switch holds identically for all three schemes. The implementation must still never comma-join the rule for any scheme.
- **DNS-at-proxy for HTTP/HTTPS proxies (the leak question).** The property claimed: with an HTTP/HTTPS proxy configured, Chromium sends the destination HOSTNAME to the proxy (absolute-form request target `GET http://host:port/path` for `http://` destinations, `CONNECT host:port` for `https://` destinations) and does **not** resolve it locally — the same no-local-DNS-leak property SOCKS5 has via its domain-name address type. Electron's docs do not state this at the wire level (as with SOCKS5 in §22), so per this project's verify-don't-assume standard it is **empirically confirmed by the e2e harness**, not asserted from docs: `tests/e2e/support/fake-http-proxy.ts` records the destination host exactly as it arrived on the wire, and `tests/e2e/proxy.spec.ts` navigates to a reserved `*.invalid` hostname (RFC 2606 — guaranteed never to resolve via a real resolver) with the HTTP proxy enabled and asserts the proxy received the literal unresolved name `http-test.invalid`, never a pre-resolved IP. If the navigation reaches the proxy at all, the name cannot have been resolved locally. **This is the gate condition from the task: if Chromium had instead pre-resolved the hostname, the `.invalid` navigation would fail with a DNS error and the proxy's request log would be empty — the test would fail rather than silently ship a leak.** (The e2e is run under xvfb by the maintainer, not in this sandbox; this note records the design and the assertion, and the CI run is the proof.)
- **Scheme-aware health probe (ADR 0012 / decision 4 generalized).** `probeProxy(config)` dispatches by scheme: `probeSocks5` (unchanged RFC 1928 method negotiation) for `socks5`; `probeHttpProxy` for `http` (plain TCP, send a `CONNECT amnesic-probe.invalid:443` and confirm the reply is an `HTTP/1.x` status line — proves the endpoint speaks the HTTP CONNECT-proxy protocol, not merely that a port is open; it deliberately does not prove the proxy can reach the wider internet, since the probe target never resolves); `probeHttpsProxy` for `https` (the connection to the proxy is itself TLS, so the probe uses `node:tls.connect` — confirmed present/reachable on electron@43's Node runtime — then the same CONNECT/status-line check over TLS). The HTTPS probe uses `rejectUnauthorized: false` **only on the throwaway probe socket**: it checks reachability + that the endpoint speaks HTTP-over-TLS, exactly as the SOCKS5 probe checks protocol without validating a credential — it does not and cannot weaken Chromium's own certificate validation on the real proxied traffic. This is a documented weaker check, not a false claim of full validation.
- **SOCKS4 excluded (not a verification gap — a deliberate honesty decision).** ADR 0007 decision 3 already documents that SOCKS4 has no domain-name address type and requires a pre-resolved IPv4, leaking every hostname to the local resolver. ADR 0012 keeps it out of the scheme list for that reason; `validateProxyConfig` rejects `socks4`/`socks`/anything not in `{socks5,http,https}`, unit-tested in `tests/unit/proxy.test.ts`.
- Source: https://www.electronjs.org/docs/latest/api/structures/proxy-config (proxyRules grammar, scheme prefixes, proxyBypassRules); `node_modules/electron/electron.d.ts` (pinned electron@43.0.0, `ProxyConfig` shape); the e2e harness described above for the DNS-at-proxy assertion.
