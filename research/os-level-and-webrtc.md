# Safe Browsing and WebRTC leak mitigation — verified against Electron 43.0.0

**Date:** 2026-07-01
**Pinned version:** electron@43.0.0 / Chromium 150.0.7871.46

## 15. `--disable-features=SafeBrowsing` — is Safe Browsing actually reachable in stock Electron?

**Verified: TRUE the feature name is real, but FALSE/overstated that it's meaningfully "phoning home" in a default Electron app — this is the most important correction in this research pass.**

Two separate things are being conflated in the plan, and they matter for how you word the README's threat-model claims:

1. **Compile-time:** Electron's own build system compiles Chromium with Safe Browsing code present by default. Evidence: `electron/electron` issue #43191 ("electron 31 does not build with `safe_browsing_mode=0`, regression from 30") shows that (a) Electron's build previously supported compiling Safe Browsing _out_ entirely via the `safe_browsing_mode=0` GN arg, (b) that capability broke as of Electron 31 and was closed "not planned" by maintainers — meaning **current Electron (including 43) ships with Safe Browsing code compiled in**, and there is no supported build-time way to strip it out yourself without patching Electron's own build config.
2. **Runtime/functional:** Safe Browsing (and other Google services bundled in Chromium — geolocation, spelling/suggest, translate) require a `GOOGLE_API_KEY` to actually authenticate and make network calls to Google's backend. Electron does **not** ship with a baked-in Google API key (unlike Google's own Chrome builds, which are signed with Google-internal keys at build time). Multiple long-standing Electron issues (#8212, #7306, #6648) confirm this: without an app developer explicitly setting `process.env.GOOGLE_API_KEY` before app start, calls to these Google services fail/no-op silently. **This means in a stock `electron@43.0.0` app that never sets `GOOGLE_API_KEY`, Safe Browsing's actual network calls to Google are non-functional even though the code path is compiled in.**

**Net verdict:** `--disable-features=SafeBrowsing` is a real, valid Chromium feature name and costs nothing to include as defense-in-depth (it also suppresses any download-protection UI/local hash-matching logic that doesn't need network access). But the plan's framing — "Chromium's built-in Safe Browsing service sends hashes of visited URLs to Google by default... otherwise your 'no record anywhere' claim is false the moment it's enabled" — overstates the risk for an Electron app specifically, since there's no functioning credential for it to phone home with unless the project explicitly adds a `GOOGLE_API_KEY`. **Correction for the README/threat-model doc:** state clearly that (a) the switch is applied as defense-in-depth, (b) the project must never set `GOOGLE_API_KEY`, and (c) this is why — not because the switch alone is a complete guarantee. This is a more defensible and more accurate claim than what's currently written.

- Sources: https://github.com/electron/electron/issues/43191, https://github.com/electron/electron/issues/8212, https://github.com/electron/electron/issues/7306, https://github.com/electron/electron/issues/6648

## 16. WebRTC leak mitigation

**Verified: the plan's claim that `--disable-webrtc` isn't real is TRUE. Its suggested fallback (permission handler + content blocking) is incomplete — there is a better, still-current, documented API it's missing.**

- Confirmed: there is no `--disable-webrtc` Chromium switch, and Chrome removed the old `chrome://flags` UI toggle for disabling WebRTC entirely years ago. No build flag or `disable-features` entry fully removes the WebRTC API surface in stock Chromium/Electron 43. This part of the plan's claim is correct.
- **What the plan is missing:** Electron has a real, current, documented API for exactly this purpose that isn't mentioned in the plan at all: **`webContents.setWebRTCIPHandlingPolicy(policy)`**, with valid values including `'default'`, `'default_public_and_private_interfaces'`, `'default_public_interface_only'`, and `'disable_non_proxied_udp'`. Setting `'disable_non_proxied_udp'` on every window's `webContents` (e.g., in an `app.on('web-contents-created', ...)` hook) forces WebRTC to route only through a configured proxy or not connect via UDP at all — the standard, current mitigation for the "WebRTC reveals real IP behind a VPN/proxy" leak class. This is a `webContents`-level method, not a `session`-level one (correction versus how such APIs are usually assumed to live on `session`).
  - This does **not** fully disable WebRTC (peer connections, data channels, media capture APIs are all still present in the page's JS environment) — it only constrains IP-handling/ICE-candidate behavior. For actually blocking WebRTC outright (matching the plan's stated goal of "not using video calls, so disable entirely"), the realistic current approach is still what the plan proposes as the fallback, refined:
    1. A preload script (or a `session.setPreloads`/`registerPreloadScript` injected script — note: `setPreloads`/`getPreloads` were deprecated in Electron 35 in favor of `registerPreloadScript`/`unregisterPreloadScript`/`getPreloadScripts`, per Electron's breaking-changes log, so use the new API if implementing this) that deletes `window.RTCPeerConnection`, `window.RTCDataChannel`, `navigator.mediaDevices.getUserMedia`, etc. before page scripts run.
    2. `session.setPermissionRequestHandler` to deny `'media'` permission requests, as the plan already says (covers camera/mic gUM prompts).
    3. `webContents.setWebRTCIPHandlingPolicy('disable_non_proxied_udp')` as belt-and-suspenders in case any RTCPeerConnection object survives step 1 (e.g., created before the preload script runs, or via a code path the deletion missed).
  - None of these three alone is a complete guarantee; combined, they match what privacy-focused Chromium forks (e.g., ungoogled-chromium-style mitigations) actually do, since Chromium genuinely doesn't expose a single supported "kill switch."
- Source: https://www.electronjs.org/docs/latest/api/web-contents (setWebRTCIPHandlingPolicy), https://www.electronjs.org/docs/latest/breaking-changes (preload script API rename, Electron 35.0)
- **Recommendation:** update the plan to (a) name `setWebRTCIPHandlingPolicy` explicitly instead of only "content-blocking," and (b) note it's a `webContents` method scoped per-window, meaning it must be (re-)applied to every new tab/`BrowserView`/webview's `webContents`, not set once globally on the session.
